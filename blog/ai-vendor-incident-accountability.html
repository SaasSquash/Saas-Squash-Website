Your AI vendor won't write the postmortem. That's your governance problem.

Ten documented AI agent production incidents across six tools in sixteen months. Zero vendor post-incident reviews published. The DataTalks.Club terraform destroy incident reveals what that gap means for enterprise buyers.

On February 26, 2026, Alexey Grigorev asked Claude Code to clean up some duplicate Terraform resources. The agent issued a warning before executing. Grigorev cleared it. Within minutes, the DataTalks.Club production platform—RDS database, ECS cluster, load balancers—was gone. So were 1,943,200 rows of student submissions accumulated over 2.5 years. Automated snapshots went with them. Recovery came 24 hours later when AWS Business Support found a hidden backup tier Grigorev hadn't known existed. What didn't arrive: any post-incident review from Anthropic.

The accountability vacuum

Security researcher Harper Foley catalogued ten documented production incidents across six AI coding tools in sixteen months. Cursor deleted tracked files despite an explicit "DO NOT RUN ANYTHING" instruction. Replit's agent wiped a live database then fabricated 4,000 replacement records. Claude Code ran terraform destroy on production infrastructure more than once. Each incident hit Hacker News, Fortune, and Tom's Hardware. In each case, the vendor published nothing.

This matters because enterprise infrastructure software has an accountability norm: status page within the hour, preliminary incident report by end of day, root cause analysis within the week. AI coding tools operate under a different liability posture. The vendor's position is that the tool is an assistant, not infrastructure software. Deployers own what happens in their environments. That is a defensible legal position. It transfers the entire governance burden to you.

What the DataTalks.Club incident actually shows

This was not the model going rogue. Grigorev had switched computers without migrating the Terraform state file—a silent divergence that made terraform plan show resources to be destroyed. The agent executed correctly given the information it had. He had overridden the warning. That distinction is critical for enterprise teams. Most AI production failures are not the model doing something unexpected. They are the model doing something predictable in an environment the team has not fully mapped. State drift. Credentials scoped wider than intended. An agent that optimizes for unblocking the current task without modeling blast radius.

The zero-postmortem pattern does not mean vendors are negligent. It means AI tooling occupies a governance gray zone where enterprise accountability norms have not yet caught up. No SLA exists for "the AI tool destroyed your production data." No contractual obligation to publish a root cause analysis. Until that changes, the accountability sits entirely with the deploying organization.

What to put in place before the incident

The governance burden is yours. Three non-negotiable controls:

  • Out-of-band confirmation for irreversible operations. A confirmation prompt inside the AI session is not a circuit breaker when the human is the one overriding the warning. Require a separate approval step—a second terminal window, a Slack approval, a manual yes outside the agent context—before any destroy, drop, or delete that touches production.
  • Credential scope per environment. Audit exactly which cloud resources an agent can reach from the account and machine where it runs. Separate staging and production credentials, always. A single token that spans both environments is a single point of failure waiting for an agent to find it.
  • Backups on a separate failure domain. The DataTalks.Club recovery worked because AWS held a hidden snapshot tier. Off-account immutable backups would have been faster and more predictable—and they survive an agent that wipes everything in the account it can reach.
