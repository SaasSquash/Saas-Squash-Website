You didn't write that code: the LiteLLM supply chain breach

In late March, Mercor—a $10 billion AI recruiting startup—was compromised not through its own application, but through LiteLLM, an open-source library sitting quietly in its dependency tree. The entry point wasn't the model. It was the plumbing.

On March 31, 2026, Mercor disclosed a breach that exposed personal records for more than 40,000 contractors, proprietary source code, and recorded video interviews. Mercor's own application code was never directly touched. The attacker—later claimed by extortion group Lapsus$—inserted malicious code into LiteLLM, a Y Combinator-backed open-source library that provides a unified interface for routing calls across major language model providers. Mercor was a downstream consumer. Once the poisoned library executed inside their stack, credentials were captured and internal systems were exposed. The window of compromise was measured in hours, but the data was already gone.

A trusted dependency became the attack surface

LiteLLM is not an obscure package. It is deployed across thousands of production AI stacks—startups, mid-market SaaS platforms, and enterprise teams alike use it to normalize calls to OpenAI, Anthropic, Google, and others behind a single API. That ubiquity is exactly what made it valuable to attackers. One insertion point, many downstream victims.

This is not a novel attack category. SolarWinds, XZ Utils, and the npm ecosystem have taught this lesson before. What is new is the privilege level that AI frameworks typically hold. LiteLLM and its peers sit between your application and your model provider. They handle API keys, authentication tokens, and the raw content of prompts and completions—including anything a user typed or a model returned. A compromised AI framework is not just code execution; it is access to the most sensitive data flowing through your product.

The Mercor breach also exposed something structural: video interviews conducted through an AI-evaluated hiring pipeline. The AI layer was the product, and the framework powering it became the liability.

Why AI tooling is an under-scrutinized attack surface

Most B2B engineering teams apply rigorous vendor review to databases, identity providers, and payment processors. Few apply the same scrutiny to AI libraries. The reasons are understandable: the ecosystem moved fast, the libraries feel lightweight, and the trust assumption is that an open-source project with GitHub stars and VC backing is safe. The Mercor incident refutes that assumption directly.

Consider what a typical AI framework can reach in your stack:

  • Model provider API keys — often with no per-request spend limits, capable of running up large bills or exfiltrating prompt data.
  • User-submitted content — documents, queries, uploaded files that pass through the framework before reaching the model.
  • Internal retrieval systems — RAG pipelines often connect the AI framework directly to databases or vector stores without a separate access boundary.
  • Output post-processing — whatever the model returns passes back through the same library, making it a natural interception point for exfiltration.

LLM providers run at 99–99.5% uptime, meaningfully below general cloud infrastructure. That brittleness pressures teams to adopt routing and fallback libraries quickly—often faster than security reviews can keep pace.

What to do before your next deployment

Mercor is a cautionary tale, not a death sentence for AI frameworks. The pattern is manageable with the same controls that protect any critical dependency—applied consistently. A practical checklist:

  • Pin versions and review changelogs before every update to AI framework packages. Treat a minor version bump the same way you would treat a dependency change in your auth library.
  • Scope API key permissions narrowly. Model provider credentials passed to a framework should have the minimum permissions required—read-only where possible, no admin access, hard spend caps enforced at the provider level.
  • Isolate the AI layer at the network boundary. The process or container running your AI framework should not have direct access to your primary data stores. Introduce an explicit service boundary with its own auth.
  • Audit what your framework touches at rest. Does it log prompts or completions? Where? Who can read those logs? Could a malicious version silently ship them elsewhere?
  • Monitor for anomalous model API calls. Sudden spikes in token usage, unexpected model endpoints, or calls at unusual hours are signals worth alerting on—not just cost tracking.
  • Subscribe to security advisories for every AI library in your stack, not just the headline dependencies. LiteLLM had no equivalent of an npm audit workflow pushing the compromise to the surface before Mercor was hit.
